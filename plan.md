## 总体规划说明

本规划基于《AI Agent 框架开发需求文档》，按「版本阶段 × 模块维度」拆解实现路径。整体分为三个主版本：

- **V0.1 MVP**：最小可用闭环（已基本完成，用于验证核心架构）。
- **V0.2 核心增强版**：补齐多模型/配置化/可观察性/基础 RAG 等核心能力。
- **V0.3 扩展与生态版**：插件、事件、CLI、多模态、高阶中间件与文档/社区建设。

每个版本内再按模块拆解为具体任务，建议在 issue 中进一步细化到工程粒度。

---

## V0.1（MVP）回顾与收尾

> 目标（对应文档 2.3、9.1）：可运行、可集成、易扩展的最小闭环。

- **已实现核心能力（代码已落地，仅列关键项）**
  - `model`：统一 `Model` 接口、OpenAI 文本模型适配器、Function Calling 桥接。
  - `agent`：`Agent` 接口 + 默认对话 `ChatAgent`（短期记忆）。
  - `memory`：`Memory` 接口 + `BufferMemory` 实现。
  - `middleware`：`Handler` / `Middleware` / `Chain` + 日志 / panic 恢复中间件。
  - `tool`：工具定义、注册表、示例工具 `calculator_add`。
  - `config`：`Config` + `FromEnv()`，环境变量加载核心配置。
  - `cmd/demo` & `cmd/tool_demo`：文本对话 Demo 和工具调用 Demo。

- **V0.1 剩余可选收尾事项**
  - **测试补强**
    - 为 `agent.ChatAgent` 增加单元测试（含有/无记忆、模型出错场景）。
    - 为 `tool.Registry` 和 `ChatWithTools` 增加基本集成测试（可通过 fake model 或打桩实现）。
  - **HTTP Demo（简单版）**
    - 新增 `cmd/http_demo`：
      - `POST /chat`：请求体 `{ "message": "..." }`，返回 `{ "reply": "..." }`。
      - 只调用默认对话 Agent，不引入复杂路由/认证。

## V0.2 核心增强版规划

> 目标：从「可跑」走向「可用」，覆盖多模型、配置化、基础 RAG、可观察性等主要功能需求。

### 1. 模型集成与多模型策略（F1.1–F1.3，部分 F1.5）

- **1.1 模型接口扩展**
  - 在 `model` 包中补全接口：
    - `Generate(ctx, prompt, opts...)`（单轮生成）。
    - `Embed(ctx, texts)`（向量化，用于后续向量记忆）。
  - 为不支持的模型能力提供合理的默认行为（如返回明确错误）。

- **1.2 模型工厂（F1.2）**
  - 新增 `model.Factory`：
    - 支持根据配置（如 `"openai/gpt-4"`, `"claude-3-opus"`）返回对应适配器。
    - 预留注册钩子（后续插件/扩展可注入）。
  - 支持最少 2 家厂商：
    - OpenAI（已有）+ 至少一家国内（如通义或文心，一期只做文本）。

- **1.3 多模型策略（F1.3）**
  - 在 `agent` 增加一个「策略选择层」：
    - 简单策略：`by name` / `by cost tag` / `by latency tag`。
  - 在配置中允许为 Agent 指定：
    - 默认模型、多模型列表、选择策略。

### 2. Agent 能力增强与记忆体系完善（F2.x）

- **2.1 Agent 类型扩展（F2.2）**
  - 在 `agent` 包中新建子实现：
    - ReAct Agent：简单版（LLM + 工具调用循环，带 step 限制）。
    - Plan-and-Execute Agent：将任务拆成子任务列表，再逐个执行（可先用 prompt 规划）。

- **2.2 记忆体系扩展（F2.5）**
  - `memory` 包新增：
    - 向量记忆接口：
      - `Search(ctx, query, k)` 等。
      - 结合 `model.Embed` 与向量库客户端（先选 1 个：如 `pgvector` 或 `Chroma`）。
    - 摘要记忆：
      - 实现一个「定期总结历史对话」的组件（基于 LLM 调用）。
  - 记忆管理器：
    - 管理短期 Buffer + 长期向量 + 摘要三种记忆的写入与读取策略。

- **2.3 输出解析器（F2.7）**
  - 实现通用的「结构化输出」工具：
    - 通过约束 prompt + JSON 校验，将 LLM 输出解析为结构体。
  - 提供内置解析器（JSON / KV / 表格），并定义扩展接口。

### 3. 中间件与可观察性增强（F3.x, F5.x, NF1–NF2）

- **3.1 中间件完善**
  - 在 `middleware` 包新增：
    - 缓存中间件（基于 key 的简单 in-memory cache，可预留 Redis 实现接口）。
    - 限流中间件（令牌桶/漏桶，先做简单按 IP/用户 ID 限流）。
    - 内容安全中间件（关键词过滤 + 简单脱敏规则）。
  - 支持中间件排序/优先级配置（在注册接口中增加 `Order` 或使用链构建器）。

- **3.2 可观察性（F5.1–F5.3，NF1–NF2）**
  - `observability` 包引入：
    - 统一日志接口（对接标准日志 & 结构化日志实现）。
    - Prometheus 指标导出：
      - 请求量、错误率、延迟直方图、token 消耗等。
    - OpenTelemetry Trace：
      - 为一次 Agent Run 创建根 Span。
      - 对关键阶段（模型调用、工具调用、向量检索）打子 Span。
  - 中间件级别集成上述能力，确保按文档的 P0 性能目标进行初步压测。

### 4. 配置化与预置 Agent 模板（F4.x）

- **4.1 配置文件加载（F4.2）**
  - `config` 包新增：
    - `Load(path string) (Config, error)`：支持 YAML/JSON。
    - 结构中包含模型、记忆、中间件、工具等配置。
  - 支持多配置文件（如 dev/stage/prod）与环境变量覆盖。

- **4.2 预置 Agent 模板（F4.4）**
  - 在 `agent` 层提供模板构造函数：
    - 对话 Agent（当前已有，增强与配置打通）。
    - 文档问答 Agent（RAG）：
      - 需要结合向量记忆实现：文档导入 -> 向量化 -> 检索 + 生成。
  - 为这两类模板各自提供一个 Demo（命令行或 HTTP）。

### 5. 非功能需求与文档（NF3–NF5）

- **5.1 文档**
  - 在当前 `README.md` 基础上：
    - 增加英文版 README。
    - 新增 `docs/` 目录：
      - `quickstart.md`、`concepts.md`、`api-reference.md`、`best-practices.md`。
  - 目标：新用户 15 分钟内跑通 RAG 或对话 Agent Demo。

- **5.2 错误处理与可用性（NF2, NF5）**
  - 统一错误类型与返回结构，区分：
    - 用户错误（4xx 风格）、系统错误（5xx 风格）、外部依赖错误（模型/DB）。
  - 在关键路径增加 panic recover，并考虑优雅关闭（signal.Notify）。

## V0.3 扩展与生态版规划

> 目标：在稳定核心的基础上，增强生态能力和高级特性，使框架适合更复杂的生产使用场景。

### 1. 多模态支持与高级模型集成（F1.4–F1.6）

- **1.1 通用多模态 Message 结构**
  - 在 `model` 中将 `Message` 扩展为：
    - 支持 `[]ContentPart`（text/image/audio 等），适配不同厂商要求。
  - 在各模型适配器内部完成格式转换。

- **1.2 多模态模型适配器**
  - 为 OpenAI（GPT-4V）、Claude-3 Vision 等提供多模态调用封装。
  - 在 Agent 模板中增加「多模态理解 Agent」示例。

- **1.3 本地模型支持（F1.6）**
  - 提供 Ollama / Llama.cpp 客户端：
    - 定义统一 LocalModel 接口。
    - 在模型工厂中通过配置启用（如 build tags 控制）。

### 2. CLI 工具与开发者体验（F4.5, NF3, NF7）

- **2.1 CLI 工具（独立 `cli` 包 + main）**
  - 功能：
    - `sath init`: 初始化一个新项目骨架。
    - `sath demo`: 运行内置示例（对话/文档问答等）。
    - `sath serve`: 启动 HTTP 服务形式的 Agent。

- **2.2 文档与社区**
  - 完善贡献指南（CONTRIBUTING）、行为准则（CODE_OF_CONDUCT）、Issue/PR 模板。
  - 在 README 中补充 Roadmap 与版本策略说明（语义化版本）。

### 3. 插件系统与事件钩子（F6.1–F6.3）

- **3.1 插件系统（编译期为主）**
  - 设计一个「插件注册中心」：
    - 插件通过匿名导入触发 `init()` 注册。
    - 插件可注册模型适配器、工具、中间件、事件监听器等。
  - 文档中明确当前阶段不支持运行时热插拔。

- **3.2 事件系统**
  - 在 Agent 生命周期关键点发布事件：
    - 初始化、开始运行、收到模型响应、执行工具、完成、异常等。
  - 提供同步/异步监听器实现，允许扩展日志、审计、通知等。

### 4. 可观察性完善与运维能力提升（F5.4–F5.5, NF1, NF6）

- **4.1 健康检查与调试模式**
  - 在 HTTP 服务模式中：
    - 暴露 `/health` 端点：
      - 返回模型 API、向量库、数据库等依赖组件状态。
    - 提供调试模式开关：
      - 输出更详细的调用栈、原始请求/响应（注意脱敏）。

- **4.2 测试与性能**
  - 单元测试覆盖率 >= 85%，对关键路径做基准测试（benchmark）：
    - 重点关注中间件链开销、工具调用开销、向量检索延迟。
  - 对 100 QPS 场景做基本压测，验证 NF1 指标是否达标。

## 建议的执行顺序（高层甘特）

- **阶段 1（当前 → V0.1 收尾，约 1–2 周）**
