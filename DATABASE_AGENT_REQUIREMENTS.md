# 数据查询 Agent 需求文档

> 本文档描述**数据查询 Agent** 的产品与功能需求，适用于多种数据存储（如 MySQL、Hive、Elasticsearch、MongoDB、Redis 等），不涉及具体实现细节。支持**只读**与可选的**可写/可修改**能力；写操作需**权限校验**与**用户确认**。

---

## 1. 概述

### 1.1 产品定位

**数据查询 Agent** 是一种专用对话助手，根据用户的自然语言描述，在对**各类数据存储**进行**只读查询**的基础上，可扩展支持**可写、可修改**操作；在保证安全的前提下完成：理解意图 → 探索数据源结构 → 编写请求（查询或写/改）→ 在满足权限与确认前提下执行 → 解读结果并以自然语言回答。

- **只读**：查询、检索、统计等不改变数据的操作，可直接执行（在系统允许的只读边界内）。
- **可写/可修改**：插入、更新、删除、结构变更等会改变数据的操作，必须在**通过权限校验**且**获得用户（或授权方）确认**后才可执行。

不同数据源使用各自的「结构概念」与「查询/写入语法」，Agent 通过统一的工作流（探索 → 编写 → 校验与确认（写时）→ 执行 → 解读）适配不同后端。

### 1.2 核心价值

- 用户用自然语言描述意图，无需手写 SQL、DSL、脚本或命令。
- 自动探索当前数据源的结构，再编写请求，降低对 schema、mapping、数据形态的记忆要求。
- **只读**默认安全、无额外确认；**可写/可修改**通过权限校验与确认机制，在可控前提下支持数据变更。
- 与现有 Agent 能力（执行管道、可观测性等）保持一致，便于集成与运维。
- 设计上可支持多种数据源与「只读 + 可选写/改」能力，由配置决定是否开放写/改及对应的权限与确认策略。

## 2. 功能需求

### 2.1 角色与能力

- **角色**：专业的数据操作助手，能针对当前所连接的数据源类型给出正确、安全的查询与（在开放时）写/改建议；对写/改类操作应明确区分「建议内容」与「已确认执行」。
- **推理方式**：采用「思考 → 行动 → 观察」的循环，根据当前对话与工具反馈决定下一步动作，直至给出最终答案。
- **能力边界**：使用与「当前数据源的结构与数据操作」相关的工具；若仅开放只读则仅使用只读工具；若开放写/改，则写/改类工具在执行前必须经过权限校验与确认流程。

### 2.2 所需能力

Agent 依赖以下**能力**（具体以何种工具、何种语法提供由实现按数据源决定）：

| 能力             | 说明                                                         | 用途                         | 执行前提           |
|------------------|--------------------------------------------------------------|------------------------------|--------------------|
| 列举数据对象     | 列出当前库/实例下的可操作对象（表/集合/索引/键空间等）       | 了解数据源中有哪些对象       | 无                 |
| 查看结构/映射    | 获取指定对象的结构信息（列与类型 / mapping / 文档形态等）   | 编写正确的请求               | 无                 |
| 执行只读请求     | 使用该数据源的只读语法执行请求并返回结果                     | 查询、统计、检索             | 无（在只读边界内） |
| 执行写/改请求    | 使用该数据源的写语法执行插入、更新、删除或结构变更等         | 在用户确认后变更数据         | **权限校验 + 确认** |

- **只读**：关系型 SELECT、ES search/aggregation、Mongo find/aggregate、Redis 只读命令等，执行前无需额外校验与确认（仍受系统只读策略约束）。
- **可写/可修改**：关系型 INSERT/UPDATE/DELETE/DDL、ES index/update/delete、Mongo insert/update/delete、Redis 写命令等，执行前必须**先校验权限**，再**获得用户或授权方确认**，通过后方可执行。

仅「与当前数据源、当前会话权限与配置」相符的工具应被纳入；执行层对只读与写/改应有明确区分，写/改路径必须强制执行权限校验与确认。

### 2.3 系统行为约定

系统应向模型提供稳定的「角色与规则」说明，且可根据数据源类型与是否开放写/改做差异化描述，至少包含：

1. **推荐工作流**  
   - 理解需求 → 探索「可操作对象」与「结构/映射」→ 按该数据源语法编写请求。  
   - **若为只读请求**：直接执行 → 根据结果回答。  
   - **若为写/改请求**：先提交「将要执行的操作描述」（如语句摘要、影响范围说明），等待**权限校验通过**且**用户/授权方确认**后再执行 → 根据执行结果回答。

2. **请求规范**  
   - 使用当前数据源的标准语法与最佳实践；只读请求注意过滤、聚合、排序与结果数量限制；写/改请求应在确认前向用户清晰说明「做什么、影响哪些对象/数据」。

3. **单步与格式约束**  
   - 每轮只执行一个步骤；必须先掌握结构再编写请求；输出需包含「思考」与「行动」的明确格式，并在得到执行结果（或确认结果）后再进行下一步。  
   - 写/改类行动在未通过权限校验与确认前，不得真正执行；系统应能区分「提议写/改」与「已确认执行」。

4. **示例**  
   - 提供针对当前数据源类型的典型示例（只读：列表、统计、聚合；若开放写/改：带确认的插入/更新/删除示例），引导模型按步骤调用工具并最终给出自然语言答案。

上述约定通过系统级提示在会话中固定生效；切换数据源或只读/写改策略时，提示中的能力范围与安全约束应随之切换。

### 2.4 执行与结束

- **执行**：在单次会话中循环「思考 → 选择工具并执行（或提交确认）→ 观察结果」，由模型决定继续或结束。
- **写/改时的额外步骤**：当模型发起写/改类操作时，系统应先进行权限校验；若通过，再发起用户（或授权方）确认；仅在确认通过后执行该操作并将结果返回给模型；若权限不通过或用户拒绝，则返回明确原因，不执行。
- **结束条件**：模型明确给出最终答案，或达到系统设定的最大步数；达到后返回该轮结果给用户。
- **结果形式**：以自然语言文本形式返回最终答案或错误说明，必要时可包含对查询/执行结果的概括或表格化展示。

### 2.5 安全需求

- **只读操作**  
  - **执行层**：仅接受该数据源定义下的只读操作（如 SQL 的 SELECT、ES 的 search、Mongo 的 find、Redis 的只读命令等），拒绝任何写操作与结构变更，并在拒绝时返回明确错误。  
  - **提示层**：系统提示中明确只读边界，与执行层策略一致。

- **可写/可修改操作**  
  - **权限校验**：在执行任何写/改请求前，必须进行**权限校验**（如：当前用户/角色是否有权对该数据源、该对象执行该类写/改）。校验不通过则拒绝执行并返回明确原因，不暴露敏感信息。  
  - **用户确认**：权限通过后，必须**获得用户或授权方的明确确认**（如：展示将要执行的操作描述与影响范围，用户确认后再执行）。未确认不得执行；取消或超时视为不执行。  
  - **执行层**：仅当「权限校验通过」且「已获得确认」时，才执行写/改请求；执行结果应可追溯（谁、何时、执行了什么、是否经确认）。  
  - **提示层**：系统提示中明确写出「写/改操作需权限与确认」「未确认不执行」等约束，与执行层策略一致。

- **安全一致性**  
  执行层与提示层对「只读边界」与「写/改的权限与确认」的约束必须一致，且与当前数据源类型及会话配置匹配。

### 2.6 非功能需求

- **可观测性**：支持执行链路追踪、步骤级与整体耗时、成功/失败统计；对写/改操作支持审计信息（如是否经过确认、执行结果）。
- **扩展性**：新增某数据源的只读或写/改工具时，应能被 Agent 识别并使用；支持新数据源时，主要增加该数据源对应的「列举 + 结构 + 只读执行」及（可选）「写/改执行」工具、权限与确认策略及配套提示即可。
- **依赖**：依赖大模型推理、会话记忆、工具注册与执行能力；数据源连接、权限模型与确认方式由部署/配置提供，可支持单数据源或多数据源、只读或只读+写改按会话/配置选用。

## 3. 使用方式

### 3.1 配置与输入

- **配置**：使用前需完成目标数据源的连接、只读（及可选写/改）工具的注册、大模型与会话等基础能力配置；若开放写/改，需配置**权限校验方式**与**用户确认方式**（如谁有权确认、确认界面或接口、超时与取消策略）。若支持多数据源，需能指定或识别当前会话使用的数据源类型与只读/写改策略。
- **输入**：用户以自然语言提出需求（只读如「列出所有表」「某表有多少条记录」；写/改如「在表 X 中插入一条记录」「把某字段更新为某值」「删除某条件的数据」等）。

### 3.2 输出与体验

- **输出**：系统返回自然语言答案；若涉及写/改，在用户确认前应清晰展示「将要执行的操作」与影响范围，确认后返回执行结果或错误说明。
- **体验**：用户无需手写语法即可完成查询与（在开放且确认后）数据变更；写/改前有明确的权限与确认步骤，避免误操作。

---

## 4. 依赖与约束

- **数据源能力**：每种数据源需能提供「可列举的操作对象」与「结构/映射信息」；若支持写/改，需能执行该数据源对应的写语法并在执行层区分只读与写/改。
- **权限与确认**：写/改能力依赖可用的**权限校验**与**用户确认**机制；实现方需定义权限模型（如按用户/角色/数据源/操作类型）与确认流程（如谁可确认、如何展示、超时与取消），并与执行层强一致。
- **工具范围**：仅将「与当前数据源、当前会话权限与配置」相符的能力暴露给 Agent；写/改工具必须在执行前经过权限校验与确认，不得绕过。
- **推理格式**：模型输出需符合「思考 + 行动」的约定格式；系统提示与示例需与该约定一致，并按数据源类型与只读/写改策略使用对应语法与示例。
- **多数据源**：同一 Agent 设计可复用于多种数据源及只读/写改策略，通过「数据源类型 + 能力集（只读或只读+写改）+ 权限与确认策略 + 对应系统提示」在配置或创建会话时选定。

- **实现形态**：采用**工具驱动（ReAct）**。模型在「思考 → 行动 → 观察」循环中显式调用列举数据对象、查看结构/映射、执行只读请求、执行写/改请求等工具，每轮单步，输出含「思考」与「行动」的约定格式；工作流（探索 → 编写 → 校验与确认（写时）→ 执行 → 解读）与安全约束（只读边界、写改权限与确认、可追溯）通过工具暴露与执行层保障实现。

## 5. 文档说明

- 本文档主要描述产品与功能需求。实现采用**工具驱动（ReAct）**形态（见第 4 节「实现形态」），需满足本需求中的工作流与安全约束。  
- **只读**为默认安全能力；**可写/可修改**为可选能力，且必须配合**权限校验**与**用户确认**使用。实现可随技术选型与代码演进调整；支持新数据源或写/改时，需在实现层提供对应能力及权限与确认流程，并同步更新本需求文档及相关实现参考。
