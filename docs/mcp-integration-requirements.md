MCP Agent 功能需求文档
1. 项目背景
本文档旨在明确MCP Agent的核心功能需求，规范Agent如何发现、集成MCP（Model Context Protocol）服务提供的能力，并在用户提问时动态调用这些能力。

2. 整体流程概述
系统主要分为两大核心阶段：Agent构建阶段 和 用户交互阶段。

构建阶段：Agent启动或初始化时，自动发现并学习所有可用的MCP工具。

交互阶段：根据用户的具体问题，智能判断并执行相应的MCP工具。

3. 详细功能需求
3.1 Agent构建阶段
在Agent服务启动或配置更新时，需执行以下初始化流程，以完成对MCP工具的“感知”和“学习”。

3.1.1 查看已配置的MCP
功能描述：读取系统配置，获取当前Agent已关联或授权的所有MCP服务列表。

输入：无（基于系统配置文件或注册中心）。

输出：一个包含多个MCP服务标识符（如服务名称、ID）的列表。

3.1.2 列举MCP能力
功能描述：针对上一步获取的每一个MCP服务，调用其 listTools 接口（或类似的功能发现接口），以获取该MCP提供的具体工具列表。

输入：单个MCP服务标识符。

输出：该MCP提供的工具列表。每个工具应包含但不限于以下元数据：

工具名称（Tool Name）

功能描述（Description）

输入参数（Input Schema）

3.1.3 能力集成至提示词
功能描述：将上一步获取的所有MCP工具信息，通过特定的格式（如JSON、函数定义等）动态添加到Agent的系统提示词（System Prompt）中。

目的：使底层大语言模型（LLM）在后续的对话中，能够“知道”当前有哪些外部工具可用，以及这些工具的功能和调用方式。

要求：提示词的构建需结构清晰，确保LLM能够准确理解并选择合适的工具。

3.2 用户交互阶段
当用户向Agent提问时，系统需根据上下文和模型决策执行以下流程。

3.2.1 意图识别与工具匹配
功能描述：LLM根据用户的当前问题、对话历史以及系统提示词中注入的MCP工具列表，判断是否需要调用外部MCP工具来完成任务。

判断依据：如果用户需求与某个MCP工具的功能描述相匹配，则模型进入工具调用流程。

3.2.2 执行MCP接口
功能描述：当模型决定调用某个工具时，系统需解析模型返回的调用指令（包括目标MCP、工具名称及参数），并实际执行对该MCP接口的调用。

输入：目标MCP标识、工具名称、参数。

处理：向指定的MCP服务发起请求。

输出：接收MCP服务的执行结果。

3.2.3 结果整合与回复
功能描述：将MCP接口返回的结果作为上下文，交还给LLM进行理解和润色，最终生成对用户的自然语言回复。

要求：确保MCP的原始数据能被LLM正确理解并转化为用户友好的回答。

4. 与现有 ReAct / tools 架构的衔接

4.1 MCP Agent 的形态
MCP Agent 本身应实现为一个独立的 ReAct Agent，遵循现有工程中的模式：

- 底层使用已有的 Tool API（tool.Registry + Execute(ctx, params)）；
- 使用 ReActAgent（或等价的基于 tools 的 Agent）作为推理/工具调用编排核心；
- 对上暴露为一个标准的 Handler（如 HTTP 路由或在其他 Agent 内部被封装为一个工具）。

换言之：

- MCP Agent 内部注册的是「访问 MCP 服务」相关的工具（例如 mcp_list_tools_xxx、mcp_call_xxx）；
- 业务 Agent（如数据查询 Agent）通过自身工具或编排逻辑向 MCP Agent 发起“子请求”，而不是自己直接持有 MCP 客户端。

4.2 工具发现结果的裁剪与命名规范

为避免将 MCP 的所有工具原样暴露给模型、导致提示过大或包含不安全能力，需对 listTools 结果进行裁剪与规范化：

1. 工具白名单 / 黑名单
- 每个 MCP 服务应支持配置：
  - 工具白名单：仅暴露配置列表中的工具；
  - 或工具黑名单：过滤掉显式禁止暴露的工具；
  - 默认策略：若无任何配置，则仅暴露显式标记为只读/安全的工具（见 4.3）。

2. 命名规范与去冲突规则
- 当同一 Agent 中集成多个 MCP 服务时，工具名必须在提示和工具注册层面保持唯一且可区分，建议：
  - 使用「服务前缀 + 点号」形式，例如：
    - vm_manager.list_instances
    - k8s.describe_pod
  - 或在工具定义中额外暴露 server 字段，由调用方工具（如统一的 mcp_call）组合 server + tool_name 完成调用。

3. 参数 Schema 的精简
- 对于远端工具暴露的完整 Input Schema，如字段过多或层级复杂，可在注入 System Prompt 时做适度精简：
  - 保留关键参数名、类型和是否必填；
  - 对于不常用或内部字段，仅在底层调用时使用，不强行暴露给模型。
- 目标是：让 LLM 能够理解「如何正确调用」而不被细节淹没。

4.3 安全策略与只读/写改区分

考虑到 MCP 工具可能包含强副作用（例如关闭服务、删除资源等），MCP Agent 在集成时需遵守以下安全策略：

1. 能力标注
- 对每个远端工具，需在本地维护或从远端元数据中推导：
  - 是否为只读工具（无外部副作用）；
  - 是否为写/改或高危操作（有外部副作用）。

2. 暴露策略
- 默认仅将「只读」工具注入给业务 Agent / 模型使用；
- 写/改类工具需要通过显式配置开启，并在 System Prompt 中明确提示其风险和使用前提。

3. 业务 Agent 一侧的控制
- 业务 Agent 在使用 MCP 能力时，应能够：
  - 选择只暴露 MCP Agent 的只读能力；
  - 若需要使用写/改能力，则结合自身已有的权限/确认机制（如 dataquery 中的“确认 token”机制），对 MCP 写操作进行二次确认或限制。

5. 非功能需求
可扩展性：支持动态增加或移除MCP服务，无需重启整个Agent核心服务（或支持热加载）。

错误处理：

在构建阶段，若某个MCP服务不可用或 listTools 失败，不应影响其他MCP服务的加载。

在交互阶段，若MCP接口调用失败（如超时、无权限），应向用户返回友好的错误提示，并可引导用户进行其他操作。

性能：构建阶段的MCP发现与能力列举过程，应尽量在后台异步完成，避免阻塞Agent的启动。

6. 数据流转图（概要）
构建阶段:
配置中心 -> Agent核心 (读取MCP列表)
Agent核心 -> MCP服务A (调用listTools)
Agent核心 -> MCP服务B (调用listTools)
Agent核心 -> 提示词构建器 (生成含工具定义的System Prompt)
提示词构建器 -> LLM上下文 (注入)

交互阶段:
用户 -> Agent接口 (提问)
Agent接口 -> LLM (用户问题 + 含工具定义的System Prompt)
LLM -> Agent核心 (决策：调用MCP工具X)
Agent核心 -> MCP服务X (执行具体接口)
MCP服务X -> Agent核心 (返回结果数据)
Agent核心 -> LLM (用户问题 + 工具返回结果)
LLM -> Agent接口 (生成最终回复)
Agent接口 -> 用户 (返回答案)
