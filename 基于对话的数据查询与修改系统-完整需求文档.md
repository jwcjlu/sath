# 基于对话的数据查询与修改系统 — 完整需求文档

**版本**：完整版（整合原需求与评审补充）  
**状态**：供评审与开发使用

---

## 1. 引言

### 1.1 背景

随着企业数据量的增长和多样化数据源的普及，业务人员和技术人员需要一种更自然、便捷的方式来访问和操作数据。传统的基于固定查询界面或手动编写 DSL（领域特定语言）的方式门槛高、效率低。本系统旨在通过自然语言对话的方式，让用户能够像与人交谈一样获取或修改数据，降低数据使用门槛，提升工作效率。

### 1.2 目标

构建一个智能对话系统，能够：

- 理解用户以自然语言提出的数据查询或修改请求。
- 自动识别并获取相关数据源的元数据信息。
- 将自然语言请求转换为对应数据库的可执行 DSL 语句（如 SQL、Elasticsearch DSL 等）。
- 连接目标数据库执行 DSL，并将结果以友好方式返回给用户。
- 支持多轮对话，能够利用上下文进行连续交互。

### 1.3 范围

本系统覆盖以下功能范围：

- 支持 MySQL、Elasticsearch、MongoDB、Redis、Neo4j、Hive 等常见数据库的查询与修改操作。
- 支持查询操作（SELECT、GET、MATCH 等）和有限的修改操作（INSERT、UPDATE、DELETE），修改操作需经过严格权限控制。
- 支持通过对话自动获取元数据（库、表、字段、索引、类型等）。
- 支持 DSL 语句的生成与执行。
- 提供对话界面（Web/App/API）供用户交互。

### 1.4 定义和缩写

| 术语     | 说明 |
|----------|------|
| DSL      | 领域特定语言，如 SQL、Elasticsearch 查询 DSL、MongoDB 查询语法等。 |
| 元数据   | 描述数据的数据，包括数据库名、表名、字段名、字段类型、索引等。 |
| 意图识别 | 从自然语言中识别用户想要执行的操作类型（查询、修改等）。 |
| 实体抽取 | 从自然语言中抽取与数据相关的实体，如表名、字段名、条件值等。 |
| 多轮对话 | 能够基于上下文进行多次交互，维持对话状态。 |
| RBAC     | 基于角色的访问控制。 |

## 2. 总体描述

### 2.1 产品视角

本产品是一个面向数据分析师、业务人员、开发者的对话式数据访问平台。用户通过自然语言输入需求，系统自动完成从理解到执行的全过程，并返回结果。系统支持多种主流数据库，能够屏蔽底层技术细节，提供统一、智能的数据访问体验。

### 2.2 用户特征

- **业务分析师**：需要快速查询数据进行分析，不熟悉数据库查询语言。
- **数据分析师**：熟悉数据但希望提高查询效率，避免手动编写复杂查询。
- **开发人员**：需要调试或检查数据，希望用自然语言代替编写临时脚本。
- **运维人员**：需要监控数据状态，快速获取关键指标。

### 2.3 假设和依赖

- 系统需预先配置好目标数据库的连接信息（地址、端口、认证信息等）。
- 用户提问时可能未明确指定数据库和表，系统依赖元数据来推断。
- 修改操作需要额外的权限审批流程（如二次确认、角色授权）。
- 依赖各数据库的驱动或客户端库进行连接和执行。

### 2.4 数据源与连接（补充）

- **数据源归属**：明确数据源是“系统级共享”还是“按用户/租户/项目”隔离；多租户或多环境（如 dev/test/prod）的隔离策略须在设计中明确。
- **连接信息存储**：连接信息（地址、端口、账号、密码等）的存储方式（配置文件、数据库、密钥管理服务）及是否加密存储须约定；仅授权角色（如系统管理员、数据 owner）可配置。
- **连接可靠性**：连接池失败重试、断线重连、健康检查策略须实现；数据源不可用时向用户给出明确提示，并可选择禁用该数据源或使用缓存元数据（仅读）。
- **数据源选择**：当用户未指定数据库或表时，系统应具备明确的选择策略：如默认数据源、最近使用数据源、或主动追问用户；多数据源时须避免歧义。

## 3. 功能需求

### 3.1 对话交互模块

- **输入方式**：支持文本输入，未来可扩展语音输入。
- **输出方式**：返回查询结果（表格、图表、文本描述）或执行状态信息。
- **多轮对话管理**：维护对话上下文，支持指代消解（如“上个月的数据”中的“上个月”需结合上下文）。
- **对话历史**：保存对话记录，便于追溯和继续对话。

### 3.2 意图识别与实体抽取

- **意图分类**：识别用户请求是查询数据、插入数据、更新数据、删除数据、获取元数据（如“有哪些表”），或对上一轮 DSL 的改写（如“把刚才的查询改成最近7天”“再查一次但只取前5条”）。
- **实体抽取**：从自然语言中提取关键信息，包括：
  - 数据库名 / 表名 / 字段名
  - 条件（比较值、范围、模糊匹配）
  - 聚合操作（求和、平均、计数）
  - 排序、分组、限制条数
  - 修改内容（插入的值、更新的字段）
- **实体不完整时的处理**：若无法识别出表名或数据库名，触发元数据获取流程，向用户提问或按既定策略自动推断。
- **多数据源消歧**：当同名表或库存在于多个数据源时，采用明确规则：提示用户选择、使用默认库、或通过命名空间/前缀约定消歧。
- **多意图与复杂句**：对单句多意图（如“查订单总量并列出前10条”）是否支持、如何拆分为多个子意图及执行顺序须在设计中明确。
- **技术选型与可解释性**：意图与实体采用规则、模板、LLM 或组合方式须约定；若需可解释性（如“为什么识别为该意图/实体”），应在交互或审计中体现。

### 3.3 元数据管理模块

- **元数据获取**：在系统启动时或用户提问需要时，从已配置的数据源拉取元数据。
  - **MySQL 等关系型**：库列表、表列表、字段名、字段类型、主键、索引。
  - **Elasticsearch**：索引列表、映射（字段类型）。
  - **MongoDB**：数据库、集合、文档结构（抽样推断）。
  - **Redis**：键的模式、类型（需配置）。
  - **Neo4j**：标签、属性、关系类型。
  - **Hive**：库、表、分区、字段。
- **元数据缓存**：缓存元数据以减少频繁拉取，支持手动刷新。
- **元数据同步策略**：定期同步的周期（如按小时/天）、增量与全量策略须明确；同步失败时具备告警与降级策略（如使用旧元数据并提示、或禁止对该数据源发起新查询）。
- **元数据辅助意图理解**：当用户提问包含模糊字段时，利用元数据中的字段名、常用值进行匹配和提示。

### 3.4 DSL 生成模块

- **NL2DSL 转换**：基于意图识别结果和元数据，将自然语言转化为对应数据库的 DSL（MySQL：SQL；ES：JSON DSL；MongoDB：查询语法；Redis：命令；Neo4j：Cypher；Hive：Hive SQL）。
- **语法校验**：对生成的 DSL 进行基本语法检查，确保可执行。
- **语义校验**：除语法外，执行前宜进行语义校验（如字段存在性、类型匹配、表关系等）；校验失败时向用户返回明确反馈，便于修正表述或补充元数据。
- **执行前确认**：对于修改类操作，应在执行前向用户展示生成的 DSL 或自然语言解释，并可展示影响行数、变更前后预览等，用户确认后再执行；支持确认超时与撤销。
- **复杂查询边界**：多表 JOIN、子查询、窗口函数、存储过程等的支持范围与优先级须明确；首版不支持的复杂特性应在约束条件中列出。
- **多语句与事务**：复杂请求可能生成多条 DSL；须明确执行顺序、是否在同一事务中执行、以及部分失败时的回滚策略（按数据库能力区分，如不支持事务的数据源仅能做单条回滚或提示）。

### 3.5 数据执行模块

- **连接管理**：管理各数据源的连接池，支持多数据源并发执行；支持配置连接池参数（最大连接数、超时等）。
- **执行引擎**：接收 DSL 语句，在对应数据源上执行。
- **权限控制**：对修改操作进行权限验证，记录操作日志；权限模型（如 RBAC/ABAC）、粒度（库/表/行/列）及配置责任方（管理员/数据 owner）须明确。
- **只读与执行限制**：支持“只读数据源”或“只读用户”；单次查询行数及结果大小上限须配置，防止误操作拖库或影响系统稳定性。
- **执行隔离**：不同用户/会话之间的执行在连接、资源与并发上应隔离，避免相互影响。
- **事务管理**：对于需要事务的操作（如跨表更新），根据数据库能力决定是否开启事务。
- **执行超时控制**：设置执行超时时间，防止长时间查询影响系统；支持异步执行模式，避免阻塞对话。

### 3.6 结果呈现模块

- **格式化输出**：根据查询结果类型，以表格、JSON、文本等形式展示。
- **可视化**：对于聚合数据，可生成简单图表（如柱状图、折线图）；须明确图表类型、生成方（前端/后端）及数据量过大时的展示策略。
- **大数据量策略**：结果过大时采用分页、采样、流式返回或“仅统计”等策略，避免一次性拉取过多数据。
- **结果导出**：是否支持将结果导出为 CSV/Excel 及单次导出大小限制须约定。
- **结果缓存**：相同或等价查询的短期缓存策略（TTL、失效条件、是否按用户隔离）可在设计中定义，以减轻数据库压力。
- **错误提示**：当执行失败时，返回用户友好的错误信息，并建议修正。

### 3.7 多轮对话与会话管理

- **上下文维护**：记录当前对话中已提及的数据库、表、条件等，用于后续指代。
- **指代消解范围**：明确指代消解的技术范围（如仅时间、仅表名、或跨句实体）及规则与模型的分工。
- **会话管理**：会话归属（用户/设备）、会话过期与清理策略、最大轮数或单会话时长须定义。
- **澄清交互**：当系统无法确定用户意图时，主动提问以获取更多信息。
- **修改确认**：对于修改操作，要求用户二次确认后再执行；确认内容可包含影响行数、变更前后预览等；批量修改时确认粒度须明确。
- **操作类交互**：是否支持“回退到上一轮”“重新执行上一条”“取消当前执行”等须在交互设计中明确。

### 3.8 错误处理与反馈

- **意图识别失败**：提示用户重新表述，或提供示例。
- **DSL 生成失败**：返回无法生成查询的原因，可能是元数据不足或自然语言歧义。
- **执行错误**：捕获数据库异常，转换为易懂的提示。
- **元数据获取失败**：提示用户检查数据源配置，并可根据策略降级或禁用该数据源。

## 4. 非功能需求

### 4.1 性能

- 意图识别与 DSL 生成的响应时间应小于 2 秒（平均）；可补充 P95/P99 延迟目标。
- 执行时间取决于数据库性能，但应避免长时间阻塞，支持异步执行模式。
- 支持并发：至少 100 个同时对话（须明确指“并发会话数”还是“并发执行数”）；可补充单用户/单数据源 QPS 或并发执行上限，防止单点占满资源。

### 4.2 可扩展性

- 支持动态添加新的数据源类型，通过插件化架构集成；插件接口定义、生命周期、配置方式及与核心的版本兼容策略须明确。
- 意图识别模型应支持在线更新和优化。
- 元数据管理模块应能适应不同规模的数据源（如数千张表）。

### 4.3 安全性

- **用户认证**：系统应验证用户身份，基于角色分配数据源访问权限；与现有 SSO/LDAP/IAM 的集成方式须约定。
- **数据脱敏**：对于敏感字段（如密码、身份证），在返回结果时自动脱敏。
- **操作审计**：记录所有用户请求和执行操作，特别是修改操作；审计日志保留期限及合规要求（如等保）须明确。
- **防注入**：生成的 DSL 应使用参数化方式，避免注入风险。
- **密钥管理**：连接信息等密钥的存储与轮换策略须符合安全规范。

### 4.4 可用性

- 提供简洁直观的对话界面，支持 Web 和移动端；可约定 Web 与移动端是否共用同一后端、移动端是否有功能或结果简化。
- 支持帮助提示，如“你可以这样问我：查询昨天订单总量”。
- 系统应 7×24 小时运行，故障恢复时间不超过 30 分钟；可补充 RTO/RPO 目标、备份与恢复策略，以及故障时的降级策略（如仅读、禁用部分数据源）。

### 4.5 兼容性

- 支持主流浏览器（Chrome、Firefox、Safari）。
- 支持与常见 IM 工具（如钉钉、飞书）集成（通过 API）；集成时的鉴权流程、消息格式、回调与限频须在接口设计中明确；是否支持企业内网部署须说明。

### 4.6 可观测性

- **监控与告警**：定义关键指标（如请求量、意图识别/DSL 生成/执行耗时、错误率）；告警规则与通知方式须配置。
- **日志与追踪**：日志级别、结构化字段、请求 trace 与执行 trace 的关联须实现，便于排查问题。
- **部署与发布**：部署架构（单机/集群）、扩缩容方式、灰度发布与回滚策略须在运维方案中明确。

## 5. 数据管理

### 5.1 元数据存储

- 使用关系数据库或 NoSQL 存储已获取的元数据，定期同步更新；同步周期与增量/全量策略见 3.3。
- 存储结构应支持不同数据库类型的元数据模型。

### 5.2 对话日志

- 存储用户对话记录、系统响应、执行结果，用于分析和模型优化。
- 遵守数据隐私法规，对日志中用户个人信息进行脱敏处理。
- 须明确日志保留期限、存储位置及合规要求；用户是否有权删除或导出自己的对话记录应在产品策略中约定。
- 涉及数据跨境、行业监管（如金融、医疗）时，须满足相应约束。

## 6. 接口定义

### 6.1 用户接口

- **Web 界面**：提供聊天窗口、输入框、结果展示区。
- **API 接口**：提供 RESTful API 供外部系统调用，接收自然语言，返回执行结果；须提供请求/响应示例、鉴权方式（如 API Key、OAuth2）、限流与配额、以及 API 版本策略。

### 6.2 数据库接口

- 各数据源需提供 JDBC/ODBC 驱动或原生客户端库。
- 支持配置连接池参数（最大连接数、超时等）。

### 6.3 外部集成（补充）

- **REST API**：鉴权、限流、版本策略见 6.1。
- **钉钉/飞书**：鉴权流程、消息格式、回调与限频、企业内网部署支持见 4.5。
- **Web 与移动端**：是否共用后端、移动端限制见 4.4。

---

## 7. 约束条件

- 修改操作仅限于已授权的数据源，且需用户二次确认；权限模型与确认细节见 3.5、3.7。
- 对于不支持事务的数据库（如某些 Redis 模式），修改操作可能无法回滚。
- 系统初始版本可能不支持所有数据库的全部特性（如复杂 JOIN、子查询、窗口函数、存储过程）；首版支持范围应在 MVP 与分期规划中明确。
- 自然语言理解能力受限于训练数据，初期可能需要用户遵循一定的表述模式。

---

## 8. 验收与演进

### 8.1 验收标准

- 定义可量化指标：如意图识别准确率、DSL 生成正确率、端到端用例通过率等。
- 建立人工抽检与回归测试流程，确保关键场景稳定可用。

### 8.2 MVP 与分期规划

- **首版（MVP）**：明确首版支持的数据库类型（如先 MySQL、Elasticsearch）及操作类型（如先只读或只读+受限修改）；不支持的复杂特性在约束条件中列出。
- **后续分期**：按阶段新增数据源（如 MongoDB、Redis、Neo4j、Hive）及能力（如更多图表、导出、IM 深度集成等）。

### 8.3 与现有系统的关系

- 明确与现有 BI、报表、数据平台的边界与集成方式（替代、互补或嵌入），避免功能重叠与数据不一致。
